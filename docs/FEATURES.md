# 기능 정리와 구현 원리 — 웹 / 하드웨어

이 문서는 현재 코드에 **실제로 구현되어 있는 것**만 정리한다. 기능마다
"무엇을 하는가 → 어떤 원리인가 → 어디에 있는가 → 어디까지가 한계인가" 순서다.
발표에서 무엇을 시연할지는 §5에 따로 있다.

---

## 0. 먼저 — 시스템이 어떻게 생겼는지

```
 ┌─ 휴대폰 (팔 끝에 장착) ─────────────────────────────┐
 │  카메라 영상 ─┬─▶ JPEG 압축 ──────────┐             │
 │               └─▶ MediaPipe Tasks     │             │
 │                   (손·얼굴·어깨 인식)  │             │
 │                        └─ 좌표만 ──┐  │             │
 │  마이크 ─▶ 브라우저 STT             │  │             │
 │  화면 ─▶ 일정·스캔·조명 UI          │  │             │
 └─────────────────────────┬───────────┼──┼─────────────┘
                    /ws/camera (양방향, 하나의 소켓)
 ┌─────────────────────────┴───────────┴──┴─────────────┐
 │  PC (Flask 서버)                                      │
 │   좌표 → 3D 월드 변환 → 추종 판단 → 관절각 계산        │
 │   Gemini 호출(문서 프레임 통째로 전달) · 일정 저장     │
 └──────────┬──────────────────────────┬────────────────┘
       시리얼(DYNAMIXEL)            HTTP
       로봇팔 서보 5개          라즈베리파이 → 서보 → 벽 스위치
```

**이 구조의 핵심 한 줄:** 인식(ML 추론)은 전부 휴대폰이 하고, 서버는 **좌표 계산과
판단만** 한다. 서버에 mediapipe(Python)가 없다 — 이유는 §4.

# 1. 웹 쪽 기능

## 1.1 모바일 계기판 (`/mobile`)

**무엇을** — 팔 끝 휴대폰에서 여는 화면. 영상 송출, 대화, 문서 스캔, 일정, 조명이
전부 여기 모여 있다. 좌: 들은 말/답변, 우: 카메라 화면.

**원리** — 페이지가 열리면 `getUserMedia`로 카메라를 잡고, 캔버스에 그린 뒤
`toBlob`으로 JPEG를 만들어 웹소켓으로 계속 보낸다(약 20fps). 보내기 전에
**방향 보정**(기기 회전각만큼 회전 + 좌우 미러링)을 캔버스 단계에서 끝낸다 —
서버가 받는 프레임과 인식 좌표계가 같아야 뒤의 모든 계산이 어긋나지 않는다.

카메라·마이크는 보안 컨텍스트에서만 열리므로 서버가 자체 서명 인증서로 HTTPS를 켠다
(`ssl_context="adhoc"`). 브라우저가 경고를 한 번 띄우는 건 이 때문이다.

**위치** — `src/templates/mobile.html` 1번 절, `src/api/camera.py`

## 1.2 음성 대화

**무엇을** — 마이크 버튼을 누르면 계속 듣는다. 5초간 말이 없으면 그때까지 들은
문장을 확정해 Gemini에게 묻고, 답을 소리내어 읽어준 뒤 자동으로 다시 듣기로 돌아간다.
손을 쓰지 않고 대화가 이어진다.

**원리** — 인식은 **브라우저 내장 `SpeechRecognition`**이다(서버로 오디오를 보내지
않는다). 침묵 5초를 타이머로 재서 문장 경계를 잡고, 답변은 `speechSynthesis`(한국어)로
읽는다. 읽는 동안에는 인식을 멈춘다 — 안 그러면 자기 목소리를 다시 듣는다.

> 별도로 `Camera`가 WebM 오디오 클립을 받아 Gemini STT로 넘기는 경로도 살아 있다
> (매직 바이트로 JPEG/WebM 구분). 지금 화면은 브라우저 STT를 쓰므로 이 경로는 예비다.

**위치** — `mobile.html` 4번 절, `src/api/gemini.py`

## 1.3 음성 명령

**무엇을** — 들은 문장이 알려진 명령이면 Gemini에 묻지 않고 그 기능을 바로 실행한다.

| 말 | 동작 |
|---|---|
| "불 켜줘 / 꺼줘" | 조명 |
| "스캔해줘", "문서 읽어줘/파싱해줘" | 문서(라이브 프레임 통째로 읽기) |
| "카메라 꺼줘 / 켜줘" | 카메라 |
| "요약 모드로 바꿔", "대화 모드로" | 답변 모드 |
| "다시 읽어줘", "그만/멈춰" | TTS 제어 |
| "내일 세시에 회의 추가", "오늘 일정 뭐 있어", "이번 주 일정" | 일정 |
| "대화 종료" | 세션 종료 |

**원리** — 정규식과 부분 문자열 매칭이다. LLM 의도 분류가 아니다. 순서가 중요한데
**조명 → 일정 → 문서 → 카메라** 순으로 먼저 걸러내고, 어디에도 안 걸리면 그때
Gemini로 넘긴다. "일정"처럼 여러 갈래에 걸리는 단어가 있어서 순서 자체가 우선순위다.

**한계** — 표현이 조금만 달라도 못 알아듣는다("불 좀 어떻게 해줘" → Gemini로 감).
발표에서는 표에 있는 문장 그대로 말하는 게 안전하다.

**위치** — `mobile.html` 7번 절

## 1.4 문서 스캔과 읽기

**무엇을** — "문서 스캔" 버튼(또는 음성 "스캔해줘", 가위 제스처)을 누르면 지금
라이브로 들어오고 있는 카메라 프레임을 **통째로** Gemini에게 보내 본문을
읽어 전문 또는 요약으로 돌려준다.

**원리** — 별도의 문서 사각형 검출·원근 보정·선택 단계는 없다. 예전엔
OpenCV(Canny 에지 → 윤곽선 검출 → 사각형 근사 → 원근 보정)로 문서를 찾아
잘라낸 뒤 그 영역만 읽혔지만, 실사용에서는 이 검출 단계 없이 라이브
프레임을 그냥 보내는 것으로 충분해 걷어냈다 — Gemini가 배경의 잡다한
글자는 무시하고 본문만 읽도록 프롬프트(`GeminiConst.DOC_PARSE_INSTRUCTION`)
에서 지시한다.

**위치** — `src/api/scan.py`(`POST /api/scan/parse` 하나뿐), `mobile.html`의
"문서 스캔" 절

## 1.5 일정 (`/calendar`)

**무엇을** — 한 달 달력 전용 화면. 날짜를 누르면 그날 일정이 뜨고 거기서 추가·삭제.
음성으로도 추가·조회·삭제된다.

**원리** — **날짜 해석은 폰이, 저장은 서버가** 한다. "내일", "다음 주 화요일",
"삼월 오일" 같은 상대 표현을 `YYYY-MM-DD`로 바꾸는 일은 전부 폰의 로컬 시계로
처리하고, 서버는 확정된 문자열만 받는 순수 저장소다 — 서버와 폰의 시간대가 달라도
"오늘"이 어긋나지 않는다. 저장은 `calendar_events.json`에 임시파일→rename으로
원자적으로 쓴다(쓰는 도중 죽어도 파일이 깨지지 않는다).

달력은 항상 **6주 42칸 고정**으로 그린다. 달마다 칸 수가 달라지면 넘길 때 높이가
출렁이기 때문. 그 42칸 범위를 `?from=&to=`로 한 번에 받아온다(칸마다 42번 부르지 않는다).

**위치** — `src/api/calendar.py`, `src/templates/calendar.html`

## 1.6 조명 제어

**무엇을** — 말 한마디나 버튼으로 방 조명을 켜고 끈다. 손바닥(보) 제스처로도 꺼진다.

**원리** — 스마트 전구 API가 **아니다.** 라즈베리파이에 붙은 서보가 **벽에 달린 물리
스위치를 눌렀다 뗀다**(`press` = 목표 각도로 갔다가 원위치, 그 뒤 PWM 신호를 끊어
떨림과 발열을 막는다).

폰은 파이를 직접 부르지 않는다. 페이지가 HTTPS인데 파이가 HTTP라 브라우저가 혼합
콘텐츠로 **조용히 차단**하기 때문이다. 그래서 폰은 항상 데스키 서버를 부르고, 서버가
파이에 전달한다.

서버가 파이에 전달하는 경로가 **둘**이고 위에 있는 것부터 시도한다.

### ① 역방향 연결 — 어느 망에서든 동작 (권장)

```
파이 ──(밖으로 나가는 웹소켓)──▶ 데스키 서버 /ws/pi   [계속 열어 둠]
폰 ──HTTPS──▶ 데스키 서버 /api/light ──(그 연결로 밀어 넣음)──▶ 파이 ──PWM──▶ 서보
```

**파이가 서버로 먼저 접속해 둔다.** 나가는 방향 연결은 웹 브라우징과 같아서 NAT를
그냥 통과한다 — 공유기 포트포워딩도, 고정 IP도, DDNS도 필요 없고 파이를 인터넷에
노출하지도 않는다. 파이가 집 와이파이에 있든 발표장 와이파이에 있든 상관없다.

**이 프로젝트가 이미 쓰는 방식과 같다.** 폰도 서버가 폰을 부르는 게 아니라 폰이
`/ws/camera`로 접속해 두고 서버가 그 연결로 제스처 명령을 밀어 넣는다. 파이가 같은
신분이 된 것뿐이다.

부수 장치들:
- **토큰 인증** — 파이는 `DESKY_PI_TOKEN`을 들고 접속해야 받아준다. 서버에 토큰이
  설정돼 있지 않으면 아예 거절한다("설정을 깜빡했는데 그냥 동작해 버리는" 쪽이 더 위험).
- **자동 재접속** — 끊기면 1초 → 2초 → … 30초로 간격을 늘리며 다시 붙는다. 파이가
  서버보다 먼저 켜져도 알아서 붙는다(시연장에서 흔한 순서).
- **핑** — 조명을 켜지 않는 동안은 트래픽이 없어서 NAT가 유휴 연결로 보고 끊는다.
  서버가 25초마다 핑을 보내 유지한다(카메라 소켓은 프레임이 계속 흘러 해당 없음).
- **id 매칭** — 명령마다 번호를 붙여 결과와 짝지으므로 요청이 겹쳐도 섞이지 않는다.

### ② 직접 호출 — 같은 내부망에서만 (폴백)

```
폰 ──HTTPS──▶ 데스키 서버 ──HTTP──▶ 파이(192.168.x.x) ──PWM──▶ 서보
```

파이가 서버에 접속해 있지 않을 때 예전 방식으로 `PI_URL`에 직접 쏜다. 서버와 파이가
같은 공유기 아래 있으면 설정 없이 예전처럼 동작한다.

> **지금 어느 경로인지 확인**: `GET /api/light/status`의 `route`가 `"link"`면 ①,
> `"http"`면 ②. `GET /api/pi/status`로 파이 접속 여부만 따로 볼 수도 있다.
> 시연 직전 점검용으로 쓸 것.

**한계** — 서보는 위치를 되읽을 수 없다. 그래서 `state`는 "서버가 마지막으로 명령한
상태"이고, 손으로 직접 스위치를 누르면 어긋난다.

**위치** — `src/api/light.py`, `src/api/pi_link.py`, `raspberry-server/server.py`

## 1.7 PC 대시보드 (`/`)

**무엇을** — 관절 슬라이더, FK/IK 계산, 3D 미리보기, 카메라 화면. 개발·정비용.

**원리** — FK/IK와 3D 렌더링은 하드웨어가 없어도 돈다. 슬라이더를 움직이면
`/api/fk`가 위치를 계산하고 `/api/render`가 **데스크톱 시뮬레이터와 같은 그리기
코드**로 PNG를 만들어 돌려준다(`kinematics/simulate.py`의 `draw_pose`를 공유).

화면 맨 위에는 **폰에서 입력할 접속 주소**가 크게 뜬다. 서버는 `0.0.0.0`에 열려 있어
같은 공유기의 기기는 이미 접속 가능한 상태인데, 정작 그 주소를 알 방법이 없었다 —
`fundamental/network.py`가 이 PC의 사설 주소를 찾아 콘솔 배너와 이 카드에 함께 띄운다.
랜카드가 여러 개일 때는 "바깥으로 나갈 때 쓰는 랜카드"를 커널에 물어 가장 유력한 것을
맨 앞에 둔다(UDP 소켓 connect 트릭 — 패킷은 실제로 나가지 않는다).

**위치** — `src/templates/index.html`, `src/api/arm.py`, `src/render.py`, `fundamental/network.py`

# 2. 하드웨어 쪽 기능

## 2.1 기구학 — FK / IK

**무엇을** — 관절각 5개 ↔ end-effector 위치를 서로 변환한다.

**원리**
- **FK**: URDF에 적힌 링크 변환 행렬을 순서대로 곱한다. 순수 파이썬 4x4 행렬
  연산이라 외부 수학 라이브러리가 필요 없다.
- **IK**: **damped least squares**(감쇠 최소자승). 자코비안의 유사역행렬에 감쇠항을
  넣어 특이점 근처에서 해가 발산하지 않게 한다. 5자유도 팔로 3자유도 위치 목표를
  풀면 해가 무수히 많으므로, **현재 자세를 시드로** 넣어 지금 자세에서 가까운 해를
  고르게 한다 — 이게 없으면 목표가 조금만 움직여도 팔이 전혀 다른 자세로 튄다.
- 수렴하지 않으면 **아예 움직이지 않는다**(`converged=False`).

**구성의 단일 소스**는 `kinematics/configure/desky.urdf` 하나다. 링크 길이·관절 축·
한계가 전부 여기서 나오고 코드에는 하드코딩이 없다.

**위치** — `kinematics/kinematics.py`, `urdf_loader.py`

## 2.2 자기충돌 안전 범위와 결합 관절

**무엇을** — 명령을 보내기 전에 그 각도가 팔이 자기 몸에 부딪히는 자세인지 검사한다.

**원리** — 서보의 물리 범위(0~300°)와 **실제로 안전한 범위는 다르다.**
`kinematics/find_joint_limits.py`가 MuJoCo로 관절을 쓸어가며 자기충돌이 시작되는
지점을 찾아 `joints.json`에 적어 두고, 런타임에는 그 값을 쓴다.

까다로운 건 **결합 관절**이다. joint2의 안전 범위가 joint3의 현재 각도에 따라
달라진다. 그래서:
- UI 슬라이더에는 상대 관절이 어디 있든 항상 안전한 **보수적 범위**를 보여주고,
- 실제 명령 직전에는 **이번에 함께 명령될 값들로** 결합 한계를 다시 푼다.

관절 하나만 움직일 때도 현재 자세에 그 값만 갈아끼운 벡터를 만들어 검사한다.

**위치** — `src/arm_service.py`의 `servo_degs_within_limits`, `kinematics/kinematics.py`의 `Joint.bounds`

## 2.3 서보 제어

**원리** — DYNAMIXEL Protocol 1.0 시리얼 통신. 눈에 안 띄지만 중요한 최적화가 둘 있다.

1. **속도가 직전과 같으면 다시 쓰지 않는다.** 매번 쓰면 관절당 시리얼 왕복 1회와
   25ms 대기가 추가되는데, 관절 5개면 그것만으로 추종 루프가 멈춘다.
2. **관절각 읽기를 캐시한다.** 한 번 읽을 때 서보 5개에 각각 왕복이 발생한다.
   통신에 **실패했을 때도 타임스탬프를 갱신**하는 게 핵심 — 안 그러면 오류가 이어지는
   동안 매번 "캐시 만료"로 보여 즉시 재시도하는 뜨거운 루프가 된다.

**위치** — `hardware/actuator.py`, `hardware/controller.py`, `src/arm_service.py`

## 2.4 얼굴 추적

**무엇을** — 사람 얼굴을 화면 중앙에 두도록 팔이 따라간다.

**원리** — 네 단계다.

1. **인식(휴대폰)** — MediaPipe FaceLandmarker가 얼굴을 찾고, 필요한 **3점만**
   보낸다: 코끝, 양쪽 눈 바깥쪽 끝.
2. **거리 추정** — 두 눈 바깥쪽 끝 사이의 **실제 길이를 안다고 가정**하고
   (성인 평균값) 화면에서 그게 몇 픽셀로 보이는지로 핀홀 공식을 역산한다.
   `거리 = 실제길이 × 초점거리 / 화면상길이`. 캘리브레이션한 값이 아니라 **추정치**다.
3. **월드 좌표 변환** — 현재 관절각으로 FK를 풀면 end-effector의 4x4 변환 행렬이
   나온다. 카메라가 거기 달려 있으므로 그 행렬이 곧 "카메라가 어디서 어디를 보는가"다.
   화면 좌표 + 거리를 이 행렬로 곱하면 월드 좌표가 된다.
4. **추종 명령** — 화면 중앙에서 벗어난 양(dx, dy)을 두 축으로 나눈다:
   **좌우(dx)는 1번 관절 yaw 회전**, **상하(dy)는 높이(z) 이동**. 좌우를 x/y 이동이
   아니라 회전으로 흡수하는 게 이 팔의 설계다(그래서 IK 목표의 x, y는 ±5cm로 clamp).

**위치** — `perception/face_tracker.py`, `perception/camera_geometry.py`

## 2.5 몸(어깨) 추적 — 얼굴 인식 실패 대비

**무엇을** — 얼굴이 안 보이거나 **멀 때** 양쪽 어깨를 대신 따라간다.

**원리** — PoseLandmarker의 33점 중 **어깨 2점(11, 12)만** 쓴다. 거리 추정은 얼굴과
같은 방식이되 기준자가 어깨 너비(40cm)다. 어깨 신뢰도가 낮으면(가려짐) 폰이 그
프레임의 몸 자체를 보내지 않는다.

전환 규칙이 이 기능의 핵심이다:

- 얼굴까지 **1m보다 가까우면** 얼굴을 추적
- **1m보다 멀면** 어깨를 추적 — 멀어지면 얼굴은 화면에서 작아져 랜드마크 잔떨림에
  약해지고 깊이 추정도 부정확해지는데, 어깨는 크고 안정적으로 잡힌다
- 얼굴이 멀리 보이는데 어깨가 안 보이면 그래도 얼굴을 쓴다(아무것도 안 쓰는 것보다 낫다)

**히스테리시스가 걸려 있다.** 경계값 하나로 부등호 비교를 하면 거리 추정치가 1m
근처에서 맴돌 때 프레임마다 판정이 뒤집혀 팔이 위아래로 훑는 진동이 생겼다. 그래서
경계를 둘로 벌렸다 — 얼굴을 보는 중이면 1.15m보다 멀어져야 몸으로, 몸을 보는 중이면
0.85m보다 가까워져야 얼굴로 돌아온다.

**위치** — `perception/body_tracker.py`, `perception/follow_controller.py`

## 2.6 손 추종 (현재 꺼져 있음)

**원리** — 손목~엄지 CMC 길이(3.5cm)를 기준자로 거리를 추정하고, 손 위치로 팔을
움직인다. 깊이는 **일부만** 보정한다 — 100% 따라가면 손을 향해 팔이 계속 달려든다.

**지금 상태** — `FollowControllerConst.HAND_FOLLOW_ENABLED = False`. 손 랜드마크는
계속 들어오고 **가위바위보는 그대로 동작하지만**, 팔이 손을 쫓아다니지는 않는다.
"제스처는 쓰되 팔이 손을 따라다니는 건 싫다"는 요구에 맞춘 스위치다.

## 2.7 idle 두리번거림

**무엇을** — 아무도 안 보이면 기준 자세로 돌아간 뒤 좌우로 천천히 두리번거린다.

**원리** — 상태 머신 두 단계. RETURNING(기준 자세 `(0, 0, 0.34)`로 IK 복귀) →
도착하면 LOOKING_AROUND(1번 관절을 ±40°, 6초 주기 **사인파**로 왕복). 사람이 나타나면
즉시 추적으로 돌아간다.

사인파를 쓰는 이유는 끝점에서 속도가 0이 되어 부드럽게 방향을 바꾸기 때문. 톱니파나
단순 왕복이면 끝에서 덜컥거린다.

**위치** — `perception/follow_controller.py`의 `_idle_step`

## 2.8 가위바위보 제스처

| 손모양 | 동작 |
|---|---|
| ✌️ 가위 | 문서 스캔 시작 |
| ✊ 바위 | 음성 대화 시작 |
| ✋ 보 | 카메라 끄기 + 조명 끄기 |

**원리** — ML 분류기가 아니라 **기하 계산**이다.

- 손가락이 펴졌는지: 손끝이 두 번째 관절보다 **손목에서 더 먼가**로 판정. 손목 기준
  상대 거리라 손이 기울어져 있어도(=팔이 돌아가 있어도) 같은 기준이 통한다.
- 엄지만 예외: 옆으로 벌어지므로 손목이 아니라 **새끼 MCP**를 기준으로 본다.
  덕분에 좌우 손 구분이 필요 없다.
- 화면 정규화 좌표는 가로/세로 픽셀 수가 달라 왜곡되므로 x에 종횡비를 곱해 편다.

**오작동 방지**가 절반이다:
- **같은 모양이 10프레임 연속** 잡혀야 확정(약 0.5초)
- **보만 20프레임** — 카메라를 끄면 제스처로 되돌릴 수 없으므로 일부러 더 오래
- 확정값이 **바뀌는 순간에만 1회** 실행(엣지 트리거). 계속 들고 있어도 반복 안 됨
- 어떤 동작이든 직전 실행 후 **2.5초 쿨다운**
- 프레임이 2초 이상 끊기면 확정 상태 초기화(껐다 켜면 다시 인식되도록)

**위치** — `perception/gesture.py`, `src/gesture_bridge.py`

## 2.9 추종이 흔들리지 않게 하는 장치들

추적 기능의 코드 절반은 "부드럽게 만들기"다. 발표에서 물어보면 답할 수 있게 정리해 둔다.

| 문제 | 장치 |
|---|---|
| 미세하게 계속 움직임 | **데드존** — 화면 중앙 반경 0.15 안이면 가만히 |
| 랜드마크 잔떨림 | **EMA 평활** — 오프셋을 지수이동평균으로 누름 |
| 대상이 프레임마다 바뀜 | **디바운스 0.5초** — 그만큼 이어져야 실제 전환 |
| 얼굴↔몸 경계 진동 | **히스테리시스 ±0.15m** (§2.5) |
| 명령이 너무 잦아 덜덜거림 | **명령 간격 0.15초** — 판단은 매 프레임, 명령만 제한 |
| 3D 미리보기가 병목 | 미리보기만 **0.1초 간격**으로 따로 갱신 |
| **누적 오차로 팔이 폭주** | 명령한 자세를 **캐시에 즉시 반영** (아래) |

마지막 것이 가장 까다로웠던 문제다. 추종 루프는 "현재 각도 + 이번 보정"으로 다음
목표를 계산하는데, 그 "현재 각도"를 최대 3초 묵은 캐시에서 가져오고 있었다. 그
3초 동안 이미 수십 번 명령이 나갔는데도 매번 같은 옛 기준에 보정을 얹으니 오차가
쌓여 점점 크게 흔들렸다. 해법은 하드웨어를 더 자주 읽는 게 아니라(시리얼 부하가
되돌아온다) **방금 명령한 값을 캐시에 바로 써 넣는 것**이었다. 서보가 아직 그
자세에 도달하지 못했더라도, 상대 보정을 쌓는 입장에서는 "우리가 의도한 목표"가
기준이어야 다음 보정이 정확히 얹힌다.

# 3. 웹과 하드웨어가 만나는 지점

두 계열이 공유하는 것은 **딱 두 개**다.

| 공유 자원 | 웹 쪽 용도 | 하드웨어 쪽 용도 |
|---|---|---|
| `Camera` (`/ws/camera`) | 영상·음성 송수신, 제스처 명령 전달 | 랜드마크 수신 |
| `ArmService` | 대시보드 상태 조회, 수동 조작 | 추종 루프의 명령 실행 |

둘 다 락으로 감싸져 있어 웹 요청 스레드와 카메라 루프가 안전하게 나눠 쓴다.
Flask는 **백그라운드 스레드**에서 돌고 카메라 루프가 **메인 스레드**를 차지하는데,
이건 OpenCV 창 관련 호출이 메인 스레드에 있어야 하기 때문이다(macOS 제약).

# 4. 왜 인식을 전부 휴대폰이 하는가

이 프로젝트에서 가장 중요한 설계 결정이고, 발표에서 질문이 나오면 답할 가치가 있다.

**처음에는** 서버(PC)가 mediapipe로 얼굴·손·몸을 전부 인식했다. 문제가 둘 있었다.

1. **속도** — PC 하나가 매 프레임 모델 추론 여러 개 + 3D 렌더링 + 하드웨어 제어를
   다 하니 폰이 보내는 20fps를 못 따라갔다. 로봇 반응이 느려지고 뚝뚝 끊겼다.
2. **화질** — 서버가 받는 프레임은 JPEG 압축과 다운스케일을 거친 뒤라 인식 자체가
   잘 안 됐다. 특히 얼굴이 그랬다.

**옮기고 나니** 둘 다 해결됐다. 폰은 **압축 전 원본 영상**에서 인식하므로 화질
문제가 사라지고, PC는 좌표 계산만 하므로 부하가 급감했다. 부수 효과도 셋이다.

- 네트워크로 나가는 건 **좌표 몇 개**뿐이다(손 21점, 얼굴 3점, 어깨 2점)
- 서버에서 **mediapipe 의존성이 완전히 사라졌다** — 인식 파이프라인이 표준
  라이브러리 + OpenCV/NumPy만으로 돈다
- 프라이버시 설명이 쉬워진다 — 인식 자체는 기기 안에서 끝난다

> 다만 정확히 말할 것: **JPEG 프레임 자체는 여전히 서버로 간다**(문서 스캔과
> 미리보기에 필요하다). "영상이 전혀 나가지 않는다"는 틀린 말이고,
> "인식은 기기 안에서 끝나고 좌표만 나간다"가 맞는 표현이다.

# 5. 발표 시연 — 무엇을 보여줄 것인가

## 결론부터: 이 네 개

| 순서 | 시연 | 왜 이걸 |
|---|---|---|
| 1 | **얼굴 추적 + 거리에 따른 몸 전환** | 말보다 장면이 강한 유일한 기능. 시작 5초 만에 관심을 잡는다 |
| 2 | **가위바위보 제스처** | "준비 과정이 없다"는 주장을 한 동작으로 증명 |
| 3 | **문서 스캔 → 요약 → 음성** | AI 스피커로 **원리적으로 불가능**한 일. 제품 논지의 핵심 |
| 4 | **음성으로 조명 + 일정** | 관제탑 주장의 근거. 짧게 마무리용 |

## 순서를 이렇게 짜는 이유

**1번을 맨 앞에** — 로봇이 사람을 따라 고개를 돌리는 장면은 설명이 필요 없다.
여기서 "저게 뭐지"가 만들어져야 뒤의 설명이 들린다. 앞에 슬라이드를 길게 깔지 말 것.

**3번이 진짜 무기다.** 1·2번은 "신기하다"에서 끝나지만, 종이를 책상에 놓기만 하면
로봇이 스스로 그것을 보고 읽어주는 장면은 "필요하겠다"로 넘어간다. **AI 스피커로
되지 않냐**는 질문은 이 장면 하나로 끝난다 — 스피커에는 눈이 없다.

**시간이 부족하면 1번과 3번만.** 2·4번은 잘라도 논지가 무너지지 않는다.

## 각 시연의 대본과 위험

### ① 얼굴 추적 (약 40초)

> 아무도 없을 때 팔이 두리번거리는 걸 먼저 보여준 뒤 → 프레임 안으로 들어간다 →
> 팔이 즉시 따라온다 → 좌우로 천천히 걸어본다 → **뒤로 물러난다**(얼굴에서 어깨로
> 전환되며 계속 따라옴)

- **위험**: 조명이 어두우면 인식이 불안정하다. 발표장 조명 확인 필수.
- **위험**: 배경에 다른 사람 얼굴이 있으면 그쪽으로 갈 수 있다. 카메라 뒤에 사람이
  서지 않도록.
- **대비**: 로컬 3D 미리보기 창을 켜 두면 인식이 되고 있는지 발표자가 즉시 안다.
  단 **청중에게는 보여주지 말 것**(개발자용 화면이라 산만하다).

### ② 제스처 (약 20초)

> 손바닥을 펴서 든다 → 약 1초 후 카메라와 조명이 함께 꺼진다

- **가위(스캔)보다 보(카메라+조명 끄기)를 쓸 것.** 조명이 같이 꺼져 변화가 눈에
  확 보이고, "되돌리기 어려운 동작은 더 오래 들고 있어야 한다"는 설계 이야기를
  자연스럽게 붙일 수 있다.
- **위험**: 확정에 약 1초 걸린다. 손을 흔들지 말고 **가만히 들고 있을 것**.
- **위험**: 카메라를 끄면 제스처로 되돌릴 수 없다. 다시 켜는 건 화면 버튼이나
  "카메라 켜줘" 음성. 미리 연습해 둘 것.

### ③ 문서 스캔 (약 60초, 가장 오래 걸림)

> 종이를 카메라 앞에 든다 → "스캔해줘" → 라이브 프레임이 그대로 Gemini에게
> 읽힌다 → 요약이 음성으로 나온다

- **위험(가장 큼)**: Gemini 호출은 네트워크에 의존한다. 발표장 와이파이가 막히면
  이 시연이 통째로 죽는다. **미리 한 번 돌려보고, 안 되면 3번을 빼는 판단**을
  미리 정해 둘 것.
- **위험**: 별도 문서 검출 단계가 없으므로 종이가 화면에 크고 또렷하게 보이는지가
  전부다 — 너무 멀거나 반사광이 심하면 Gemini가 글자를 잘 못 읽는다.
- **대비**: 잘 읽히는 종이 한 장을 정해서 그것만 쓸 것. 즉석에서 아무 종이나 쓰지 말 것.

### ④ 조명 + 일정 (약 30초)

> "불 켜줘" → 서보가 벽 스위치를 누른다 → "내일 세시에 회의 추가해줘" →
> 일정 화면을 열어 등록된 걸 보여준다

- **여기서 물리 스위치라는 점을 강조할 것.** 스마트 전구가 아니라 벽 스위치를
  직접 누르므로 **집을 새로 바꾸지 않아도 된다**는 게 논점이다.
- **위험**: 음성 명령은 표에 있는 문장 그대로 말해야 한다. 즉흥적으로 바꾸면
  Gemini로 넘어가 엉뚱한 답이 나온다.

## 보여주지 말 것

- **PC 대시보드의 관절 슬라이더** — 개발 도구다. "직접 조작하는 화면"을 보여주면
  "준비 과정을 없앴다"는 논지와 정면으로 부딪힌다.
- **3D 미리보기 창** — 발표자에게는 유용하지만 청중에게는 노이즈다.
- **IK 수식이나 mediapipe 이름** — 질문받으면 답하되 먼저 꺼내지 말 것.
  청중이 기억하는 건 팔이 따라오는 장면이지 damped least squares가 아니다.

# 6. 코드에서 발견한 것 — 문서 불일치 하나

`perception/follow_controller.py`의 모듈 docstring에 이런 문장이 남아 있다.

> "몸 인식은 얼굴/손과 달리 서버에서 직접 mediapipe Pose로 돈다"

**지금은 사실이 아니다.** `body_tracker.py`, `perception_loop.py`, `mobile.html`,
`requirements.txt` 모두 몸 인식도 휴대폰으로 옮겨졌다고 말하고 있고, 서버에는
mediapipe가 설치되어 있지도 않다. 몸 인식을 마지막으로 옮길 때 이 한 줄만 남은 것으로
보인다. 발표 자료를 이 파일 기준으로 쓰면 틀린 설명이 되므로 고쳐 두는 게 좋다.
